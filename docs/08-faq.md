## 一、认证相关

### Q1: 认证失败的常见原因有哪些？

**A**: 认证失败通常由以下原因导致：

#### 1.1 网络问题
- **现象**: 认证超时、连接失败
- **排查**:
  - 检查设备是否在同一局域网
  - 检查 SoftBus 连接状态
  - 查看日志中的网络错误码
- **解决**: 确保网络连通，重试认证

#### 1.2 用户操作超时
- **现象**: 认证超时，错误码 `ERR_DM_INPUT_TIMEOUT = 96929820`
- **原因**: 
  - PIN 码输入超时（60 秒）
  - 用户确认超时
- **解决**: 提醒用户及时操作，重新发起认证

#### 1.3 HiChain 组创建失败
- **现象**: 错误码 `ERR_DM_ADD_GOUTP_TIMEOUT = 96929819`
- **原因**: 
  - HiChain 服务未启动
  - 凭证组参数不合法
- **解决**: 
  - 检查 HiChain 服务状态
  - 验证认证参数（GroupType、UserId）

#### 1.4 权限不足
- **现象**: 调用认证接口返回权限错误
- **原因**: 
  - 应用无 `ohos.permission.DISTRIBUTED_DEVICE_CONNECTOR` 权限
  - 非系统应用调用系统接口
- **解决**: 
  - 添加必要权限到 `config.json`
  - 系统接口仅限系统应用调用

#### 1.5 Session 问题
- **现象**: Session 开启超时（15 秒），错误码 `ERR_DM_CHANNLE_OPEN_TIMEOUT = 96929818`
- **原因**: 
  - SoftBus Session 创建失败
  - 对端设备未响应
- **解决**: 
  - 检查对端设备是否在线
  - 重启 SoftBus 服务

### Q2: PIN 码超时如何处理？

**A**: PIN 码超时机制设计如下：

#### 2.1 超时配置
```cpp
const int32_t INPUT_TIMEOUT = 60;  // 60 秒超时
const char* INPUT_TIMEOUT_TASK = "deviceManagerTimer:input";
```

#### 2.2 超时处理流程
1. 认证发起端展示 PIN 码
2. 启动 60 秒倒计时定时器
3. 超时后自动取消认证
4. 清理 Session 和状态机
5. 通知用户认证失败

#### 2.3 开发建议
- **UI 优化**: 显示倒计时，提醒用户及时输入
- **重试机制**: 超时后允许重新发起认证
- **日志记录**: 记录超时事件，便于排查问题

### Q3: 免交互认证（同账号）为什么失败？

**A**: IDENTICAL_ACCOUNT 免交互认证可能失败的原因：

#### 3.1 账号信息不一致
- **现象**: 仍需手动确认
- **原因**: 
  - 两设备登录的华为账号不同
  - 账号信息未同步到 DM
- **解决**: 
  - 确保两设备登录同一账号
  - 检查账号系统状态

#### 3.2 设备数量上限
- **现象**: 认证失败，提示设备数量过多
- **原因**: 同账号设备数量超过 128 台
- **解决**: 
  - 在云端设备管理中解绑旧设备
  - 删除不再使用的设备

#### 3.3 凭证组不存在
- **现象**: 免交互认证失败，退回到手动确认
- **原因**: 
  - 首次认证，凭证组尚未创建
  - 凭证组已被删除
- **解决**: 
  - 首次认证需要手动确认
  - 检查 HiChain 凭证组状态

#### 3.4 网络问题
- **现象**: 认证超时
- **原因**: 网络不稳定，无法查询账号信息
- **解决**: 检查网络连接，重试认证

---

### Q4: 认证过程中用户取消操作如何处理？

**A**: 用户取消操作的处理流程：

#### 4.1 取消触发点
- 用户拒绝认证确认框
- 用户关闭 PIN 码输入页面
- 用户主动调用取消接口

#### 4.2 取消处理
```cpp
// 用户操作返回值
// operateAction = 0 - 用户拒绝
// operateAction = 1 - 用户同意
// operateAction = 2 - 用户操作超时
```

#### 4.3 资源清理
- 关闭 SoftBus Session
- 删除认证上下文
- 停止所有认证定时器
- 通知应用认证失败

## 二、发现相关

### Q5: 为什么发现不到设备？

**A**: 设备发现失败的可能原因：

#### 5.1 设备未发布
- **现象**: 发现列表为空
- **原因**: 对端设备未调用 `PublishDevice()`
- **解决**: 
  - 确保对端设备已启动
  - 检查对端设备是否调用发布接口

#### 5.2 发现超时
- **现象**: 发现 120 秒后自动停止
- **原因**: 默认发现超时时间为 120 秒
- **解决**: 
  - 在超时时间内完成发现
  - 或调整超时时间（需修改配置）

#### 5.3 传输介质不匹配
- **现象**: 特定传输方式发现失败
- **原因**: 
  - BLE 发现：需蓝牙权限
  - WiFi 发现：需同一局域网
- **解决**: 
  - 检查蓝牙权限和状态
  - 确保设备在同一 WiFi 网络

#### 5.4 过滤条件过严
- **现象**: 发现列表为空，但对端设备已发布
- **原因**: 发现过滤条件不匹配
- **解决**: 
  - 检查 `SubscribeInfo` 过滤条件
  - 使用更宽松的过滤条件

---

### Q6: BLE 和 WiFi 发现有什么区别？

**A**: 两种发现方式的对比：

| 对比项 | BLE 发现 | WiFi 发现 |
|--------|---------|----------|
| **传输距离** | 短距离（约 10 米） | 局域网范围 |
| **功耗** | 低功耗 | 较高功耗 |
| **速度** | 较慢 | 较快 |
| **权限要求** | 蓝牙权限 | 网络权限 |
| **适用场景** | 近距离设备发现 | 局域网设备发现 |
| **穿透性** | 差（被墙壁阻挡） | 好（穿墙能力强） |

**开发建议**：
- 优先使用 BLE 发现（功耗低）
- 大范围发现使用 WiFi
- 支持混合发现（BLE + WiFi）

---

### Q7: 发现到的设备信息不完整怎么办？

**A**: 设备信息分阶段返回：

#### 7.1 发现阶段
- 返回基本信息（设备名、设备类型）
- 不包含完整设备信息

#### 7.2 认证完成后
- 返回完整设备信息（含 capabilities、deviceId 等）
- 设备信息存储在 DDK

#### 7.3 获取完整信息
- 调用 `GetTrustedDeviceList()` 获取已认证设备信息
- 调用 `GetDeviceInfo()` 获取特定设备详情

## 三、上下线相关

### Q8: 设备 ONLINE 但不 READY？

**A**: 这是正常的双层状态转换过程：

#### 8.1 状态含义
- **ONLINE**: 设备物理在线（SoftBus 连接已建立）
- **READY**: 设备逻辑就绪（DDS 信息已同步）

#### 8.2 ONLINE → READY 转换
1. SoftBus 通知设备上线
2. DM 设置状态为 ONLINE
3. DM 请求 DDS 同步设备信息
4. DDS 同步完成
5. DM 设置状态为 READY

#### 8.3 卡在 ONLINE 的原因
- DDS 同步失败
- DDS 服务未启动
- 网络问题导致同步超时

#### 8.4 解决方案
- 检查 DDS 服务状态
- 查看同步错误日志
- 重启设备或服务

---

### Q9: 设备反复上下线？

**A**: 设备状态抖动可能的原因：

#### 9.1 网络不稳定
- **现象**: 设备频繁 ONLINE/OFFLINE
- **原因**: WiFi 信号弱、网络切换
- **解决**: 
  - 改善网络环境
  - 使用有线连接

#### 9.2 SoftBus 连接问题
- **现象**: Session 频繁断开重连
- **原因**: SoftBus 心跳超时、异常断开
- **解决**: 
  - 检查 SoftBus 日志
  - 调整心跳间隔

#### 9.3 设备休眠
- **现象**: 设备周期性上下线
- **原因**: 设备进入休眠模式
- **解决**: 
  - 调整设备休眠策略
  - 保持设备唤醒

#### 9.4 状态机 Bug
- **现象**: 状态转换逻辑错误
- **原因**: 代码 Bug 或竞态条件
- **解决**: 
  - 查看状态转换日志
  - 修复状态机逻辑

---

### Q10: 离线超时是多久？

**A**: 设备下线延迟机制：

#### 10.1 超时配置
```cpp
const int32_t DEVICE_OFFLINE_DELAY = 60;  // 60 秒延迟
```

#### 10.2 延迟下线机制
1. SoftBus 通知设备物理下线
2. DM 启动 60 秒定时器
3. 定时器内设备重新上线 → 取消下线
4. 定时器超时 → 设置状态为 OFFLINE

#### 10.3 设计目的
- **容忍网络抖动**: 避免短暂断线导致下线
- **快速恢复**: 设备重连后无需重新认证
- **用户体验**: 减少频繁的上下线通知

## 四、ACL 相关

### Q11: ACL 老化到 Cache 阶段后怎么办？

**A**: ACL 三阶段老化处理：

#### 11.1 Cache 阶段特点
- SessionKey 已删除
- 仅保留凭证信息
- 需要重新认证才能恢复

#### 11.2 恢复方式
1. **免交互恢复**（Transition 阶段）
   - 2 天内重认证
   - 无需用户确认
   - 自动回到 Active 阶段

2. **完整认证**（Cache 阶段）
   - 调用 `AuthenticateDevice()`
   - 需要用户确认
   - 认证成功后回到 Active

#### 11.3 开发建议
- 检测 ACL 状态，提示用户重认证
- 优先尝试免交互认证
- 保留原设备信息，避免重新配置

---

### Q12: 多用户切换后 ACL 状态？

**A**: 多用户场景下的 ACL 管理：

#### 12.1 用户隔离
- 每个用户独立的 ACL
- 用户切换不影响其他用户的 ACL

#### 12.2 用户切换时
- 前台用户的 ACL 生效
- 后台用户的 ACL 暂停
- 用户重新登录恢复 ACL

#### 12.3 开发注意事项
- 每次用户切换需重新检查 ACL 状态
- 不要跨用户访问设备信息
- 监听用户切换事件更新状态

## 五、开发相关

### Q13: 系统应用和普通应用的区别？

**A**: 两种应用的权限差异：

#### 13.1 权限差异
| 功能 | 系统应用 | 普通应用 |
|------|---------|----------|
| 设备认证 | ✅ | ✅ |
| 设备查询 | ✅ | ✅ |
| 设备解绑 | ✅ | ✅ |
| 代理认证 | ✅ | ❌ |
| 系统级配置 | ✅ | ❌ |

#### 13.2 系统应用识别
```json
{
  "bundleName": "com.example.systemapp",
  "apl": "system_core",  // 系统 APK 级别
  "privileged": true     // 特权应用
}
```

#### 13.3 开发建议
- 普通应用：仅使用公开 API
- 系统应用：可使用代理认证等高级功能
- 检查应用权限，避免调用非法接口

## 六、性能相关

### Q18: 认证流程耗时多久？

**A**: 认证流程各阶段耗时：

#### 18.1 耗时分解
| 阶段 | 预计耗时 |
|------|---------|
| 设备发现 | 5-10 秒 |
| Session 建立 | 2-5 秒 |
| 用户确认 | 取决于用户操作 |
| 认证执行 | 10-30 秒 |
| **总计** | **约 30-60 秒** |

#### 18.2 性能优化
- 使用免交互认证（IDENTICAL_ACCOUNT）
- 预加载设备信息
- 优化网络配置

---

### Q19: 设备数量对性能的影响？

**A**: 设备数量性能影响：

#### 19.1 理论上限
- 单用户最多 128 台设备
- 单设备最多 128 个可信设备

#### 19.2 性能影响
| 设备数量 | 性能影响 |
|---------|---------|
| < 10 台 | 无明显影响 |
| 10-50 台 | 轻微延迟 |
| 50-100 台 | 明显延迟 |
| > 100 台 | 显著卡顿 |

#### 19.3 优化建议
- 定期清理不常用设备
- 使用设备分组管理
- 限制同时活跃设备数

## 七、安全相关

### Q20: 如何保证认证安全性？

**A**: DM 的安全机制：

#### 20.1 传输安全
- 所有通信基于 TLS
- SessionKey 动态协商
- 密钥定期更新

#### 20.2 身份验证
- HiChain 凭证组验证
- 账号系统关联
- 设备证书验证

#### 20.3 权限控制
- 基于 TokenID 的权限检查
- ACL 访问控制列表
- 应用权限隔离

#### 20.4 开发建议
- 不要绕过安全检查
- 及时更新密钥
- 遵循最小权限原则
